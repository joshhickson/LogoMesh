# **Strategic Blueprint and Academic Integration Plan for LogoMesh: AgentBeats Phase 2 and NeurIPS 2026**

## **Executive Overview**

The transition of the LogoMesh architecture into Phase 2 of the AgentX-AgentBeats competition represents a critical juncture that intersects advanced software engineering with cutting-edge academic artificial intelligence research.1 Following a first-place victory in the Software Testing Agent track during Phase 1, the infrastructure must now evolve from a domain-specific evaluation tool into a highly generalized, multi-track adversarial system.1 This transformation is precipitated by severe new operational constraints, most notably the implementation of a strict four-call limit on Large Language Model (LLM) application programming interfaces within the Lambda Custom Track.1 This specific constraint functionally breaks the live deployment of the Monte Carlo Tree Search (MCTS) algorithms that powered the Phase 1 Red Agent, necessitating a rapid architectural pivot toward an offline simulation paradigm.1

Concurrently, a strategic academic partnership with Professor Tianyu Shi and researchers from McGill University introduces the opportunity to elevate the LogoMesh platform from a competition entry into a foundational dataset and benchmark suitable for submission to the Neural Information Processing Systems (NeurIPS) 2026 conference.1 Professor Shi’s leading research in Multi-Agent Reinforcement Learning (MARL), specifically his work on the DynaWeb and Cascading Cooperative Multi-agent (CCMA) frameworks, provides the exact theoretical scaffolding required to solve LogoMesh’s current engineering hurdles.1 By utilizing DynaWeb's world-model simulation techniques, the LogoMesh engineering unit can successfully execute the necessary "Offline Sandbox" pivot, allowing the Red Agent to dream and execute vast MCTS rollouts without consuming the restricted live API limits.1

This comprehensive report provides an exhaustive analysis of the AgentBeats Phase 2 landscape, a deep architectural review of the required engineering pivots, an analysis of Professor Shi’s MARL frameworks, and a structured strategic agenda to govern the impending integration meeting. The analysis dictates that aligning the engineering hurdles of the competition with the theoretical requirements of the academic research will yield both a highly competitive autonomous agent and a rigorous, publication-ready framework.

## **Comprehensive Evaluation of the AgentX-AgentBeats Phase 2 Landscape**

The AgentX-AgentBeats Phase 2 competition, designated as the "Purple Phase," fundamentally alters the operational reality established in Phase 1\.1 While the initial phase focused on the creation of "Green Agents" to serve as automated benchmarks, judges, and environment managers, Phase 2 requires the deployment of "Purple Agents" engineered to systematically engage, navigate, and defeat these established evaluation environments.1 The competition structure abandons the continuous evaluation model in favor of a highly compressed, sprint-based format spanning from March 2, 2026, to May 24, 2026\.1

The sprint schedules introduce rotating evaluation tracks that demand rapid architectural adaptation. The analysis of these tracks reveals a high degree of complexity regarding the input and output requirements of the target Green Agents. The transition from evaluating isolated Python functions to parsing multi-repository infrastructures, interacting with decentralized blockchains, and executing bash-level patches requires a robust abstraction layer.

The sprint architecture dictates the immediate development priorities for any generalized Purple Agent. The competition is structured into four distinct periods, each isolating specific domains of artificial intelligence competency.1

### **The Lambda Custom Track: Asymmetric Warfare and State Constraints**

The immediate operational priority is the Lambda Agent Security Custom Track, which enforces a highly rigid, asymmetric evaluation paradigm.1 The environment pits an Attacker file against a Defender file in an isolated execution arena entirely devoid of external internet access.1 Submissions consist of arbitrary code that is downloaded daily at 12:00 Anywhere on Earth (AoE) to run validation checks before entering an all-versus-all battle format.1

The architectural constraints placed upon these agents dictate the boundaries of the engineering effort. The Defender agent is strictly stateless. It possesses no historical memory across interaction rounds and undergoes a complete restart mechanism upon each individual request.1 Furthermore, the Defender is subjected to a mandatory "helpfulness" protocol via Normal User Tests.1 Should a Defender agent attempt to achieve perfect security by adopting an over-refusal posture—rejecting benign inputs—it will automatically fail the baseline evaluation and be disqualified from the leaderboard.1 This forces the Defender into a highly vulnerable position where it must accurately parse intent without the benefit of conversational context.

Conversely, the Attacker agent operates with full stateful memory. It is granted complete visibility into the chat history and the scenario context, allowing it to maintain an internal database of successful and failed injection vectors across multiple rounds.1 The Attacker must execute either direct attacks (manipulating the defender through direct dialogue) or injection attacks (embedding malicious instructions within seemingly benign data, such as a JSON resume file).1

The scoring relies on a Top-K matching algorithm, evaluating the win rate of all Attackers against the top ten Defenders, and vice versa, executing up to seven interaction rounds per battle.1 The true measure of agent efficacy is derived from a private leaderboard featuring entirely unseen scenarios, demanding high systemic generalization rather than programmatic overfitting to known public scenarios like the "Resume Downgrade" or "PortfolioIQ" simulations.1 Furthermore, a strict resource limit has been implemented, capping each agent at four LLM API requests per battle, fundamentally altering the viability of real-time search algorithms.1

### **Target Green Agent Complexity in Standard Tracks**

To achieve the stated goal of transforming LogoMesh into a generalized self-evaluator, the architecture must eventually interface with the Green Agents featured in Sprint 3: the Coding Agent and Cybersecurity Agent tracks.1 The Phase 1 success of LogoMesh relied on a structured JSON-RPC payload expecting defined parameters such as source code, test code, and natural language rationale.1 The target environments for Phase 2 present radically divergent communication protocols and execution requirements.

Within the Coding Agent track, the target Green Agent AgentSWE (an agentified variant of the SWE-bench Verified benchmark) requires the Purple Agent to navigate massive repository structures and output functional .patch files utilizing simulated bash and debug modes.1 Another Coding track winner, NetArena, simulates microservice network policy misconfigurations in Kubernetes, requiring iterative debugging to restore communication patterns without destabilizing the cluster.1

In the Cybersecurity track, the Ethernaut Arena benchmark demands interaction with local Anvil blockchains to audit and exploit Solidity-based smart contracts across 41 progressively difficult levels.1 The AgentSlug (RCA-Bench) target requires the root-cause analysis of C/C++ memory vulnerabilities discovered via fuzzing pipelines in the ARVO dataset.1 Meanwhile, the Chai GPT Green Agent employs an adaptive seven-agent ecosystem to benchmark security detectors against MITRE ATT\&CK techniques, generating evolving attack campaigns within secure Docker sandboxes.1

This extreme variance in domain languages (Python, Bash, Solidity, C/C++) and output formats indicates that the core LogoMesh orchestrator must be decoupled from the execution layers. The engineering roadmap must prioritize the development of a universal "Task Router" capable of dynamically parsing inputs and translating internal strategies into the specific dialect required by the target Green Agent.1

## **The Interoperability Framework: A2A Protocol and Platform Architecture**

Central to Phase 2 participation is strict adherence to the Agent-to-Agent (A2A) protocol.1 The A2A protocol is an emerging open standard designed to standardize communication between AI agents deployed across disparate systems.7 While the Model Context Protocol (MCP) focuses on connecting agents with static tools and data sources, A2A facilitates dynamic, back-and-forth collaboration in natural modalities.7

The AgentBeats platform utilizes A2A as the universal interface to decouple evaluation logic from agent implementation.8 This allows any A2A-compatible Purple Agent to be tested by any Green Agent without requiring custom integration or benchmark-specific adaptations.8 The communication mechanisms rely on standard web technologies, with messages sent via HTTP and data structured in JSON.9 Responses can be delivered via Server-Sent Events (SSE) for real-time updates, polling for periodic checks, or push notifications via webhooks for long-running tasks.9

The typical A2A interaction flow within the AgentBeats assessment paradigm involves several critical steps. First, the assessment launcher script initiates the process, ensuring the Purple Agent is reset to a clean state.5 The Purple Agent acts as the client, fetching an "Agent Card" from the Green Agent server, which contains the necessary metadata and capabilities.10 The Purple Agent then generates a unique Task ID and sends the task payload to the Green Agent.10

For LogoMesh to operate effectively across the varied tracks of Phase 2, its internal architecture must seamlessly intercept these A2A requests, decode the objective, and route the logic to the appropriate sub-agent solver (e.g., Python, Solidity, Bash) before returning a correctly formatted A2A response.1 The implementation of an A2A-compliant Task Router is a non-negotiable engineering prerequisite for generalization.

## **LogoMesh Architectural Assessment and Engineering Hurdles**

The Phase 1 architecture of LogoMesh established a highly reliable evaluation baseline by synthesizing adversarial security testing with deterministic ground-truth anchoring.1 The system deployed a Purple Agent to generate code alongside a Green Agent Assessor that utilized an embedded Red Agent, a Docker Sandbox, and a Static Analyzer.1 The evaluation produced a Contextual Integrity Score (CIS) encompassing four orthogonal dimensions: Rationale Integrity, Architectural Integrity, Testing Integrity, and Logic Score.1

By restricting LLM-based subjective adjustments to a strict boundary of ±0.10 against the deterministic sandbox outputs, LogoMesh successfully mitigated the variance and "vibes-based" scoring associated with standard LLM-as-a-judge methodologies.1 Furthermore, the system employed Intent-Mismatch Detection, utilizing cosine similarity to catch instances where an AI hallucinates or returns code entirely unrelated to the prompt, applying severe penalty multipliers.1

Baseline metrics from Phase 1 demonstrated an average CIS of 0.55 across twenty evaluation tasks.1 The architecture exhibited high precision on fundamental algorithmic challenges, such as recursive Fibonacci sequences (scoring 0.75), while accurately capturing the difficulty gap inherent in complex distributed systems like Raft consensus algorithms and Multiversion Concurrency Control (MVCC) architectures (scoring \~0.40 \- 0.49).1

### **The "Uroboros" Security Threat and Sandbox Isolation**

Despite the robust scoring mechanism, the Phase 2 kickoff analysis identified critical vulnerabilities within the physical execution environment. The most severe of these is the "Uroboros" threat.1 In the current repository state, the Red Agent logic, governed by src/red\_logic/orchestrator.py, runs in-process with the primary evaluation engine.1 Given that the Red Agent is explicitly designed to hunt for and execute malicious injections (such as SQL injections or prompt jailbreaks), running this code without strict air-gapping presents an unacceptable risk of host contamination and sandbox escape during evaluation loops.1

The immediate engineering directive requires the total containerization of the Red Agent orchestrator utilizing Docker.1 This isolation is classified as a Priority Zero (P0) objective. Without strict air-gapping, the impending mass simulation workloads required for Phase 2 will invariably trigger self-inflicted execution failures or compromise the integrity of the local evaluation databases.1 The integration of GitHub Actions for automated linting and formatting (flake8) is also required to maintain code quality as the infrastructure scales.1

### **The 4-Call LLM Limit and the "Offline Sandbox" Pivot**

The defining engineering hurdle for the impending Lambda sprint is the newly discovered constraint of a hard limit of four LLM API calls per request/battle.1 The core mechanism of the LogoMesh Red Agent relies on Monte Carlo Tree Search (MCTS).1 MCTS, by its mathematical nature, requires an extensive iterative loop of selection, expansion, simulation, and backpropagation to explore the vast action space of prompt injections and code vulnerabilities.11 Executing a meaningful MCTS algorithm is mathematically impossible within a four-call budget, as the search tree cannot achieve the required depth or breadth to escape local optima and discover complex vulnerabilities.1 Furthermore, complex tasks like those in SWE-bench natively require multi-step reasoning often exceeding twenty turns.1

This constraint mandates an immediate architectural pivot. LogoMesh can no longer operate as a live, on-the-fly MCTS engine during the competition battles.1 Instead, the entire system must be transformed into an "Offline Sandbox" testing facility.1 The engineering unit will utilize their local, unconstrained compute resources to run simulated battles against mock stateless defenders.1 The local MCTS engine will execute thousands of iterations to map the vulnerability landscape of the provided scenarios.

By analyzing the SQLite Battle Memory generated during these offline runs, the system will identify the highly optimized, exact sequence of interactions required to break the defender.1 The final submission to the AgentBeats competition will therefore not be the MCTS engine itself, but rather a highly lightweight, deterministic execution script that deploys the pre-calculated attack sequences derived from the offline simulations.1

### **DevOps Modernization and Non-Determinism Mitigation**

The post-mortem analysis of Phase 1 highlighted the severe impact of LLM non-determinism on evaluation stability.1 Successful prompt injections demonstrated highly variable success rates when re-tested. To guarantee stability, the engineering pipeline must enforce a minimum of ten to twenty local simulation rounds per scenario to filter out low-probability attack vectors before they are added to the final execution script.1

This requirement for high-velocity mass simulation is currently bottlenecked by inadequate DevOps infrastructure.1 The lack of automated preliminary checks results in engineering cycles wasted on syntax formatting, indentation errors, and flake8 violations.1 The implementation of comprehensive GitHub Actions for automated linting and Continuous Integration (CI) is a prerequisite for scaling the multi-agent simulations.1 Furthermore, the migration of project tracking to the Linear platform will resolve the organizational fragmentation that hampered the later stages of Phase 1\.1

## **Academic Synergy: Integrating Professor Tianyu Shi’s MARL Frameworks**

The integration of Professor Tianyu Shi into the LogoMesh initiative provides a critical academic dimension that directly addresses the complex engineering hurdles outlined above.1 Professor Shi, an incoming faculty member at McGill University with a Ph.D. from the University of Toronto, possesses a distinguished research portfolio centered on Multi-Agent Reinforcement Learning (MARL), Large Language Model agents, and robust decision-making under uncertainty.12 His work specifically focuses on collaborative intelligence and the application of generative modeling for complex simulations.12 An analysis of his recent publications reveals theoretical frameworks that offer elegant mathematical solutions to the empirical problems currently facing the LogoMesh architecture.

### **DynaWeb: Model-Based Reinforcement Learning via Imagination**

The most highly relevant work in Professor Shi’s portfolio is the recent publication of *DynaWeb: Model-Based Reinforcement Learning of Web Agents*.2 Online reinforcement learning for autonomous agents is historically constrained by the massive costs, inefficiencies, and risks of irreversible actions when interacting with live, real-world environments.2 DynaWeb circumvents these barriers by introducing a Model-Based Reinforcement Learning (MBRL) framework that relies on high-fidelity simulation.2

The core innovation of DynaWeb is the training of a Web World Model (WWM), denoted as ![][image2], which acts as a synthetic proxy for the real environment.4 Instead of expending expensive API calls or risking irreversible actions online, the agent policy ![][image3] interacts entirely within this simulated world model.2 The WWM processes the current observation state ![][image4] (structured as an accessibility tree) and the agent's proposed action ![][image5] to predict the subsequent state transition ![][image6].4

Crucially, to optimize computational efficiency, the WWM does not attempt to render the entire next state from scratch. Instead, it predicts a natural language description of the state changes, ![][image7].4 During inference, this delta is applied to the original state, heavily leveraging the instruction-following capabilities of the LLM.4 Furthermore, the DynaWeb framework stabilizes the learning process by interleaving fully real expert trajectories from historical training data with the on-policy imagined rollouts generated during the simulation.13 This ensures the agent benefits from high-quality demonstrations while reducing reliance on costly live exploration.13 The policy is then optimized using sequence-level policy optimization, allowing for robust credit assignment over long-horizon tasks.13

The theoretical alignment between DynaWeb and the new LogoMesh Phase 2 roadmap is exceptionally strong. The four-call LLM limit in the Lambda track requires LogoMesh to abandon live MCTS in favor of offline simulation.1 By implementing a DynaWeb-style World Model to accurately simulate the stateless Defender agent, LogoMesh can allow its Red Agent to "dream" thousands of MCTS rollouts locally without exhausting the AgentBeats platform API limits. The local SQLite Battle Memory from Phase 1 serves perfectly as the corpus of "real expert trajectories" to be interleaved with the simulated attacks, stabilizing the discovery of optimal prompt injections.

### **CCMA: Cascading Cooperative Multi-Agent Framework**

Professor Shi’s expertise extends into hierarchical coordination via the *Cascading Cooperative Multi-agent (CCMA)* framework.3 Originally designed to address the highly complex task of on-ramp merging in autonomous driving, CCMA divides the optimization problem into three distinct, hierarchical tiers: individual, regional, and global.3

The framework utilizes traditional Reinforcement Learning at the individual level to maintain high-responsiveness for immediate, micro-level interactions.3 At the regional level, fine-tuned Large Language Models act as coordinators, utilizing spatial awareness and semantic reasoning to guide individual agents toward collective goals without overwriting their local autonomy.14 At the highest tier, a Retrieval-Augmented Generation (RAG) mechanism dynamically alters the global reward functions based on real-time assessments of the overarching environment.14

This cascading architecture addresses the inherent flaw in centralized multi-agent systems, where a single orchestrator becomes a computational bottleneck or fails to generalize across diverse tasks.14 The empirical evidence demonstrates that CCMA significantly outperforms baseline models in sample efficiency and global stability.15

For LogoMesh, which currently relies on a relatively flat UCB1 multi-armed bandit for strategy selection, adopting the CCMA hierarchical approach will allow the system to scale across the diverse domains required by Phase 2\.1 The proposed A2A Task Router can be modeled as the CCMA regional LLM coordinator. When the router intercepts an A2A payload, it will utilize semantic reasoning to identify the required environment (e.g., recognizing the need for Solidity auditing for the Ethernaut Arena). It will then distribute the specific task parameters to the individual RL-driven sub-agents specializing in Python, Bash, or C/C++, while the global RAG mechanism pulls domain-specific scoring tolerances from a vector database of Phase 1 Green Agent codebases.1

## **Strategic Roadmap for NeurIPS 2026 Datasets and Benchmarks Track**

The ultimate culmination of the Phase 2 effort is the formal submission of the LogoMesh research findings to the NeurIPS 2026 conference.1 The submission deadlines perfectly map to the conclusion of the AgentBeats Sprints in early May.1 The NeurIPS submission portal opens on January 8, 2026, with the abstract submission deadline firmly set for May 11, 2026, Anywhere on Earth (AoE), followed by the full paper deadline on May 15, 2026\.16 All contributing authors must establish an OpenReview account prior to submission, and profiles must include current institutional affiliations to avoid desk rejection.16

Given the nature of the LogoMesh architecture as an evaluation arena, the research outputs are highly suited for the **Datasets and Benchmarks (D\&B) Track**.16 This track requires strict adherence to standardized data hosting, code availability, and metadata formatting to ensure the reproducibility and longevity of the submitted benchmarks.16

### **Data Engineering and Croissant Metadata Compliance**

The NeurIPS D\&B track mandates that all datasets be hosted on recognized, persistent platforms dedicated to machine learning datasets. Acceptable platforms include the Harvard Dataverse, Kaggle, Hugging Face, or OpenML.16 Bespoke hosting solutions or private Google Drive links are strictly prohibited and will result in desk rejection.16

More critically, all data must be documented using the machine-readable **Croissant metadata format**.16 Croissant is a standardized metadata vocabulary built on schema.org that extends existing approaches to describe content, provenance, and usage restrictions.16 It integrates with the Model Context Protocol (MCP) and enables fast audit and governance protocols.16

The LogoMesh engineering unit must begin structuring the output of the SQLite Battle Memory and the Docker Sandbox telemetry to automatically map to the Croissant schema.1 This involves formally defining the provenance of the generated code, the exact task configurations across the 20 base tasks, and the statistical distribution of the Contextual Integrity Scores.1 The academic team will rely on this highly structured, compliant dataset to support the mathematical claims regarding the effectiveness of the DynaWeb-enhanced MCTS algorithms.1 Failure to provide valid Croissant metadata files or to ensure that the dataset is accessible to reviewers at the time of submission will result in removal from the conference.16 Therefore, pipeline formatting must be prioritized immediately alongside the Red Agent isolation tasks.1

### **Division of Labor and Authorship Strategy**

To ensure maximal efficiency leading up to the May deadlines, strict operational boundaries must be established between the engineering and academic teams.1

The core engineering team, led by Josh Hickson, will maintain total ownership of the repository infrastructure.1 This includes managing the GitHub Actions CI/CD pipelines, containerizing the Red Agent to defeat the Uroboros threat, developing the A2A Task Router, and maintaining the stability of the Docker Sandbox.1 The engineers are responsible for ensuring that the empirical testing ground remains highly available, deterministic, and capable of generating the telemetry required for the research.1

The academic cohort, led by Professor Shi and comprising graduate students such as Yichen Shen, will assume ownership of the theoretical architecture and evaluation metrics.1 The students will interface directly with src/scoring/ and the DBOM logic, implementing the mathematical adaptations required for the CCMA routing and DynaWeb simulations.1 Their primary mandate will be translating the telemetry gathered during the competition sprints into rigorous academic prose, synthesizing the data into the required NeurIPS LaTeX templates, and conducting ablation studies to prove algorithmic efficacy.1

Finally, the meeting must address the outstanding blind spots identified in the Sprint 3 discovery phase regarding generalization.1 The team must solicit Professor Shi's guidance on designing an A2A task router capable of generalizing across extreme multi-language environments (Solidity, Java, C/C++) without collapsing under prompt-length limitations.1 The discussion should explore how to practically implement the CCMA framework's regional LLM coordinators to effectively parse and format the drastically different output requirements (e.g., bash patches versus transaction payloads) while maintaining the integrity of the underlying MCTS vulnerability hunting logic.1 Furthermore, clarification must be sought on whether external infrastructure can be utilized to host the MCTS engine if the AgentBeats platform's compute limits prove too restrictive.

## **Conclusion**

The evolution of the LogoMesh architecture for the AgentBeats Phase 2 competition requires a sophisticated synthesis of robust software engineering and advanced academic theory. The constraints imposed by the competition—specifically the severe restriction on live LLM API calls and the extreme variance in target Green Agent environments—render traditional, flat adversarial models obsolete. By executing a rapid pivot to an "Offline Sandbox" architecture and isolating the execution environment through strict Docker containerization, the platform's empirical integrity and security are secured against threats such as the in-process Uroboros vulnerability.

Furthermore, the integration of Professor Tianyu Shi’s research paradigms elevates the project beyond a mere competition entry. Applying the DynaWeb methodology allows the Monte Carlo Tree Search to function unhindered within a simulated world model, bypassing API limitations through the use of dreamed trajectories interleaved with historical expert data. Concurrently, the CCMA framework provides the necessary hierarchical routing to manage diverse coding environments ranging from Kubernetes to smart contracts. By aligning these engineering deployments with the stringent metadata, hosting, and Croissant formatting requirements of the NeurIPS 2026 Datasets and Benchmarks track, the collaboration is positioned to deliver both an apex-tier autonomous agent and a highly impactful contribution to the field of multi-agent reinforcement learning.
