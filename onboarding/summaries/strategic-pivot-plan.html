This document outlines a detailed, phased implementation plan for a strategic pivot from a rubric-based implementation to a formal, mathematically-defensible, algorithm-based model of the Contextual Integrity Score (CIS).

1. The Big Shift: From Rubric to Mathematics

üî¨ The Fundamental Change

This pivot elevates our core IP by grounding each component of the CIS in hard technology:

  • Vector-based semantic analysis for measuring intent alignment
  • Graph theory for architectural integrity
  • Semantic test coverage beyond simple line/branch metrics

This moves us from a "consulting framework" to a "hard tech" product.

2. The Three New Analyzers (v2)

Analyzer 1: Rationale Integrity (Semantic Alignment)

Old Approach: LLM-based rubric scoring

New Approach: Vector embeddings and cosine similarity

  • Generate high-dimensional vectors for requirements (text) and code (git diffs)
  • Implement function: R(Œî) = cosine_similarity(v_intent, v_code)
  • Output: Similarity score between 0 and 1

Impact: Mathematically quantifies how well code matches stated intent

Analyzer 2: Architectural Integrity (Graph Centrality)

Old Approach: Simple escomplex complexity metrics

New Approach: Directed graph validation of service architecture

  • Define project architecture as directed graph in logomesh.arch.json
  • Parse code changes to identify new cross-service dependencies
  • Implement function: A(Œî) = 0 if illegal edge detected (Critical Veto)

Impact: Enforces architectural boundaries, prevents distributed monoliths

Analyzer 3: Testing Integrity (Semantic Coverage)

Old Approach: Pass/fail capture in sandbox

New Approach: Semantic alignment + code coverage

  • Integrate standard coverage tool for line/branch metrics
  • Generate embeddings for test assertions and acceptance criteria
  • Implement function: T(Œî) = combine(coverage_score, semantic_similarity)

Impact: Detects "vanity tests" with high coverage but low business validation

3. The Golden Dataset Approach

üéØ Validation Strategy

The new analyzers are too complex for simple mock-based tests. Solution: Create a manually-labeled "Golden Dataset" of code samples with expected scores.

Required Samples:

  • High-Integrity Sample: Well-written, aligned, tested code (Expected CIS: > 0.9)
  • Rationale Debt Sample: Correct code linked to irrelevant requirement (Expected R: < 0.3)
  • Architectural Debt Sample: Code introducing illegal dependency (Expected A: 0)
  • Testing Debt Sample: 100% coverage but vanity tests (Expected T: < 0.4)

5. Why This Matters

üíé From Good to Great

This pivot transforms the Contextual Integrity Score from a qualitative assessment into a mathematically-defensible metric.

  • IP Strength: Mathematical formulas are patentable
  • Scientific Rigor: Reproducible, verifiable results
  • Competitive Moat: Far harder to replicate than rubrics
  • Product Scalability: Automated, deterministic evaluation
